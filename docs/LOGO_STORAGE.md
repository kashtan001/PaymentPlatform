# Хранение логотипов каналов

## Проблема

Логотипы каналов сохраняются в `static/uploads/channel-logos/` на локальном диске. При деплое на Railway, Heroku или аналогичные платформы **файловая система эфемерная** — при каждом редеплое контейнер пересоздаётся, и загруженные файлы теряются.

В результате:
- Ссылка на логотип в БД остаётся (`/static/uploads/channel-logos/channel-xxx.png`)
- Файл на диске отсутствует → 404
- На лендинге оплаты отображается битая картинка

## Варианты решения

### 1. Внешнее хранилище (рекомендуется)

Сохранять логотипы в S3, Cloudinary, R2 или аналогичный сервис. В БД хранить полный URL (например, `https://cdn.example.com/logos/channel-xxx.png`).

**Плюсы:** Надёжно, масштабируемо, переживает редеплои.  
**Минусы:** Требует доработки кода и настройки внешнего сервиса.

### 2. Persistent Volume (Railway)

Volume `paymentplatform-volume` монтируется в `/data`. Приложение настроено на хранение логотипов в `/data/channel-logos/`.

**В Railway Variables** сервиса PaymentPlatform добавьте:

```
UPLOADS_VOLUME_PATH=/data
```

Либо используется `RAILWAY_VOLUME_MOUNT_PATH`, если Railway задаёт его автоматически при подключённом volume.

### 3. Base64 в snapshot

При создании payment link загружать логотип, конвертировать в base64 data URL и сохранять в `snapshot_channel_logo_url`. Лендинг всегда показывает картинку из snapshot.

**Плюсы:** Не зависит от файловой системы.  
**Минусы:** Увеличение размера БД и snapshot.

### 4. Fallback (уже реализован)

При ошибке загрузки изображения (`onerror`) лендинг показывает первую букву названия канала вместо логотипа. Картинка не «ломает» интерфейс.
